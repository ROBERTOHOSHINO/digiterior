KANRI is not a new idea. Its architecture was drawn a decade ago — a platform to connect people, products, time, and value across every party in a business. What was missing was a way to build it without a large database and a large team. That is what changed. This document specifies how KANRI is built today: with an AI as its engine, and no relational database at its core.

00

設計思想 — なぜ、データベースを持たないか

KANRI has no relational database at its core. This is not a limitation. It is the design.

A traditional platform of this scope begins with schemas, servers, and a team to maintain them. KANRI begins instead with flat files and an index — plain, human-readable records, connected by a color-coded index rather than foreign keys. State lives in files; meaning lives in the index; relationships are resolved by reference, not by join.

The result is loose coupling: any part can be deployed, moved, or rebuilt without touching the rest. The same principle runs from the interface down to the authentication layer. It is also what makes KANRI buildable by a founder and an AI, and cheap to run — the architecture and the business model are the same shape.

# a resource is a flat record + an index entry — no schema, no server
people/cust_0421.json      { name, initials, history_ref, … }
index/people.idx         cust_0421 → 顧 · #1A237E · active
index/time.idx           cust_0421 → 2016-… → now → next
01

4つの資源 — the data model

From 2016 · the five actor lanes → four resources

The 2016 diagram organized a business into lanes of actors. KANRI generalizes those lanes into four resources — the four things every organization actually manages. Three are familiar. The fourth is the one other systems ignore.

People

Customers, employees, members, partners, suppliers.

Products

Goods, services, projects, intellectual assets.

Time

History, activity, schedule, future intent — as an asset.

Value

Revenue, cost, profit, payment.

Each resource is a set of flat records with an index. There is no central schema binding them — they are joined at read time through the index, which is why the model can grow one resource at a time without migrations.

02

Master Data First

KANRI begins with meaning, not process.

Master → Relationship → Knowledge → Process → Interface

Most enterprise projects fail upward: they wire processes on top of dirty master data, and the mess compounds. KANRI inverts the order. Clean, meaningful master data comes first — because it is the foundation for both human decisions and AI decisions. An agent is only as good as the master data it reasons over.

03

業務フロー — states & INDEX

From 2016 · PRE-ORDER → ORDER → 検収 → 評価

The business flows from the original diagram become explicit state machines. State transitions are recorded as dated flags in the index — the status of anything is knowable at any point in time, without a query engine.

B2B·Prospect→Estimate→Order→Delivery→Invoice→Profit
B2C·Member→Event→Participation→Subscription→Payment
# state lives as a dated flag in the index — not a DB row
index/order.idx   ord_1180 → estimate:2026-06-30 → order:07-02 → delivery:pending
04

記号インターフェース — one character, one meaning

「日本語は生まれながらにしてカラフルだった。」

KANRI's interface is built from single characters and color: 花・森・空・金・風. One character carries a meaning; a color carries a state. Together they let dense enterprise information be read at a glance on a mobile screen — the index is not just internal plumbing, it is the interface.

This is not decoration. It is information compression: the same principle that lets a haiku hold a world in seventeen syllables, applied to business data. It is also the visible surface of the authentication layer described in section 08.

05

Learning Loop

From 2016 · 評価ランク
Action → Result → Review → Learning → Improvement → Action

The 2016 diagram already ended each supply cycle in an evaluation rank. KANRI makes that loop the core mechanic: every business process is also a learning process. Each completed cycle writes back into the index as experience, so the next cycle — human or agent — starts from what the last one learned.

06

Agentic layer — Claude.ai as the engine

In KANRI, the AI is not a feature bolted on. It is the engine the system is built with and runs on.

Claude.ai reads the flat files and the index, reasons over the master data, and acts across the business flows — drafting an estimate, chasing an invoice, ranking a supplier, surfacing what a departing employee knew. The human sets intent; the agent operates on their behalf against the knowledge base.

Human → Agent → Knowledge base (files + INDEX) → Business flows → External parties

Because the knowledge base is plain files rather than a proprietary database, the agent needs no special connector to reason over the whole company. The architecture and the AI meet naturally — which is precisely why it can be built by a founder and an AI, iterating daily.

07

時間 ＝ 価値 — the resource no one manages

From 2016 · 「利益＝時間。品質＋付加価値。」

The destination was written on the original diagram ten years ago. Every system manages people, products, and value; KANRI adds the one they omit — time itself, treated as a first-class resource with history, activity, and future intent.

Managing time well is not efficiency for its own sake. It is what returns hours to creativity, relationships, and life.

This is where the platform meets the human thesis, and where KANRI becomes more than enterprise software.

08

認証インターフェース — the sealed layer

KANRI authenticates through the same symbolic surface it displays — a name, a character, a color: a mark that is public in appearance, yet cannot be reproduced.

This specification describes only the interface: how the authentication mark is presented and consumed by KANRI. The mechanism that makes the mark unreproducible — drawn from something that cannot be copied, phished, or brute-forced — is filed as patent and deliberately held as a trade secret.

What is public is only the low-value surface — like a castle's stone wall, shown openly because its depth is not reproducible. The defensibility of the platform lies in what this document does not describe.

Interface specified · mechanism sealed · patent filed, held individually
